## 1. Общее описание проекта

**Manage MegaPayment Service** — это внутренняя веб-система банка, предназначенная для наблюдения за состоянием платежных сервисов и работы с их эксплуатационной информацией.

Система не является клиентским или публичным продуктом. Она используется внутренними сотрудниками банка для работы с платежными сервисами, внутренними интеграциями и внешними платежными направлениями, например KoronaPay и другими связанными сервисами.

Основная идея проекта заключается в том, чтобы предоставить техническим специалистам единую рабочую точку, в которой можно:
- видеть список платежных сервисов;
- понимать, какой сервис работает штатно, какой деградирует, а какой недоступен;
- открывать карточку конкретного сервиса и получать связанную техническую информацию;
- использовать систему как централизованную эксплуатационную панель, а не как обычный справочник.

Проект не выполняет сами платежные операции. Его задача — помогать наблюдать за состоянием сервисов, быстрее замечать отклонения и поддерживать эксплуатационную работу с ними.

## 2. Проблема и цель проекта

В эксплуатации платежных сервисов информация часто оказывается разрозненной: часть данных может находиться в отдельных таблицах, часть — в технических заметках, часть — в системах мониторинга, а часть — передаваться между сотрудниками вручную. Это усложняет понимание текущего состояния сервисов и увеличивает время реакции на проблемы.

Основная проблема состоит в том, что техническому специалисту сложно быстро получить целостную картину:
- какие сервисы существуют;
- какие из них являются критичными;
- какие находятся в проблемном состоянии;
- где искать дополнительную информацию по конкретному сервису;
- кто отвечает за сервис и что делать при деградации.

Цель проекта — создать единую внутреннюю систему, которая объединит наблюдение за состоянием платежных сервисов и эксплуатационный контекст по ним. Система должна сократить время первичного анализа, повысить прозрачность сервисной инфраструктуры и упростить принятие правильных действий при возникновении проблем.

## 3. Контекст предметной области

Проект относится к области сопровождения и наблюдения за платежной инфраструктурой банка.

В центре внимания находятся платежные сервисы, которые могут быть:
- внутренними сервисами банка;
- интеграционными сервисами;
- внешними платежными направлениями и провайдерами;
- сервисами, влияющими на прохождение платежных операций.

Для банка такие сервисы критичны, потому что их недоступность, деградация скорости или ошибки обработки напрямую влияют на стабильность платежных операций и на работу внутренних команд.

По этой причине система должна показывать не только формальный список сервисов, но и их эксплуатационную значимость:
- к какой категории относится сервис;
- насколько он критичен;
- каково его текущее состояние;
- есть ли по нему признаки деградации;
- кто является ответственным или владельцем сервиса.

Таким образом, проект находится на пересечении двух областей:
1. каталога и структуры сервисов;
2. наблюдения за их фактическим состоянием.

## 4. Пользователи и роли

На текущем этапе проекта можно выделить следующие роли пользователей.

### 4.1. Технический специалист / инженер сопровождения
Основной пользователь системы.
Работает с состоянием сервисов, отслеживает отклонения, анализирует сигналы, открывает карточки сервисов и принимает решение о дальнейших действиях.

### 4.2. Оператор / сотрудник мониторинга
Использует систему для наблюдения за текущим состоянием сервисов, просмотра статусов, поиска проблемных сервисов и передачи информации дальше по регламенту.

### 4.3. Владелец сервиса / ответственный за сервис
Является ответственным за конкретный сервис или группу сервисов. Может поддерживать в актуальном состоянии справочную информацию, связанную с сервисом, и участвовать в разборе проблемных ситуаций.

### 4.4. Администратор системы
Отвечает за настройку системы, поддержку справочных сущностей, права доступа и административное сопровождение приложения.

На первом этапе роли можно реализовать в упрощенном виде, однако уже в описании проекта важно разделить:
- тех, кто наблюдает и анализирует;
- тех, кто сопровождает и актуализирует данные;
- тех, кто администрирует саму систему.

## 5. Что делает технарь в системе

Технический специалист не просто «смотрит статусы». Его роль в системе связана с эксплуатационной реакцией на состояние сервисов.

Базовый сценарий работы технаря выглядит следующим образом:
1. открыть систему и увидеть текущее состояние платежных сервисов;
2. обнаружить сервисы с признаками деградации или недоступности;
3. открыть подробную информацию по конкретному сервису;
4. понять контекст: категория, критичность, текущее состояние, связанная информация;
5. провести первичный анализ ситуации;
6. принять дальнейшее решение по регламенту.

В зависимости от зрелости системы дальнейшее действие может включать:
- подтверждение проблемы;
- эскалацию ответственному сотруднику или команде;
- переход к связанным инструментам анализа;
- просмотр истории изменений состояния;
- фиксацию служебного комментария;
- в будущих версиях — запуск разрешенных эксплуатационных действий.

Таким образом, технарь использует систему как рабочую панель для наблюдения, первичного анализа и выбора следующего шага, а не как обычную форму редактирования данных.

## 6. Основные сущности системы

На текущем этапе можно выделить следующий базовый набор сущностей.

### 6.1. Service
Основная сущность проекта.
Описывает конкретный платежный сервис, за состоянием которого наблюдает система.

Примеры возможных атрибутов:
- идентификатор;
- название;
- описание;
- категория;
- уровень критичности;
- текущий статус;
- внешний или внутренний тип;
- владелец сервиса;
- ссылка на дополнительные материалы.

### 6.2. Category
Категория, к которой относится сервис.
Позволяет группировать сервисы по логическому или функциональному признаку.

### 6.3. Status
Состояние сервиса.
Для реального проекта рекомендуется использовать не только `OK / NOT OK`, а более выразительную модель, например:
- OK;
- Degraded;
- Down;
- Unknown;
- Maintenance.

### 6.4. User
Пользователь системы.
Содержит информацию о сотруднике, который работает с приложением.

### 6.5. Role
Роль пользователя в системе.
Определяет, какие действия ему доступны.

### 6.6. ServiceOwner
Ответственный за сервис.
Может быть выделен как отдельная сущность или как связь между сервисом и пользователем.

### 6.7. Comment / Note
Служебный комментарий или заметка по сервису.
Может использоваться для фиксации наблюдений, уточнений и эксплуатационного контекста.

### 6.8. StatusHistory
История изменений состояния сервиса.
Позволяет понять, когда началась проблема, как часто повторялись инциденты и как менялось состояние сервиса со временем.

На старте проекта минимально достаточными являются сущности:
- Service;
- Category;
- Status;
- User;
- Role.

Остальные сущности могут быть введены по мере перехода от базовой версии к более зрелой эксплуатационной системе.

## 7. Основные сценарии использования

На первом этапе проекта можно выделить несколько ключевых сценариев использования системы.

### 7.1. Просмотр общего состояния сервисов
Пользователь открывает систему и получает общий список всех платежных сервисов с их текущими статусами, критичностью и категориями. Это позволяет быстро оценить текущее состояние инфраструктуры.

### 7.2. Поиск проблемного сервиса
Пользователь замечает деградацию или получает внешний сигнал о проблеме, после чего находит нужный сервис по названию, категории, статусу или уровню критичности.

### 7.3. Анализ конкретного сервиса
Пользователь открывает карточку сервиса и изучает его текущее состояние, историю изменения статусов, принадлежность к категории, владельца сервиса, служебные комментарии и ссылки на связанные инструменты анализа.

### 7.4. Первичная реакция на проблему
После обнаружения проблемы пользователь фиксирует факт деградации, добавляет комментарий, подтверждает необходимость реакции и принимает решение о дальнейших действиях: эскалации, переходе в смежные инструменты или наблюдении.

### 7.5. Поддержание актуальности справочных данных
Ответственный пользователь или администратор создает новый сервис, обновляет категорию, назначает владельца или изменяет служебную информацию, чтобы данные о сервисной инфраструктуре оставались актуальными.

### 7.6. Просмотр истории состояний
Пользователь анализирует, как менялось состояние сервиса во времени, чтобы понять характер проблемы, длительность деградации и повторяемость инцидентов.

Эти сценарии формируют основу первой версии и позволяют проекту быть не только справочником, но и рабочим инструментом сопровождения.

## 8. Функциональные требования

На уровне первой версии система должна обеспечивать следующий базовый функционал.

### 8.1. Работа со списком сервисов
- просмотр списка сервисов;
- отображение названия, категории, критичности и текущего статуса;
- поиск по названию;
- фильтрация по категории, статусу и уровню критичности.

### 8.2. Работа с карточкой сервиса
- просмотр полной информации о сервисе;
- отображение текущего состояния;
- просмотр владельца сервиса;
- отображение служебного описания;
- просмотр связанных комментариев и заметок;
- просмотр истории изменений статуса.

### 8.3. Работа со справочными сущностями
- создание, редактирование и просмотр категорий;
- создание, редактирование и просмотр сервисов;
- назначение владельца сервиса;
- управление базовыми статусами.

### 8.4. Работа с пользователями и ролями
- просмотр пользователей системы;
- назначение ролей;
- разграничение прав на просмотр и изменение данных.

### 8.5. Поддержка эксплуатационного контекста
- добавление служебных комментариев;
- хранение ссылок на дополнительные материалы;
- фиксация истории изменения состояния сервиса.

Функциональные требования первой версии должны быть ограничены разумным минимумом: система должна быть полезной уже на старте, но без перегрузки сложной автоматизацией.

## 9. Наблюдаемость и интеграции

Система не должна рассматриваться как самостоятельный движок мониторинга. Ее задача — использовать существующий наблюдательный контур и связывать технические сигналы с прикладной информацией о сервисах.

На уровне концепции проект может работать со следующими источниками данных:
- система метрик и статусов;
- источники алертов;
- внешние или внутренние endpoint-проверки;
- служебные ссылки на логи, трассировки и дополнительные диагностические инструменты.

В качестве основного контура наблюдаемости предполагается использование:
- OpenTelemetry для формирования телеметрических данных;
- OpenTelemetry Collector для приема и передачи данных;
- Prometheus как источника метрик и текущих состояний;
- Alertmanager как источника уведомлений о проблемах;
- blackbox-проверок для внешних или слабо управляемых интеграций.

Наш проект не дублирует эти системы, а объединяет их результаты с внутренним каталогом сервисов. Это позволяет видеть не только технический сигнал, но и понимать, к какому сервису он относится, насколько сервис критичен и кто за него отвечает.

На первом этапе интеграции могут быть реализованы в упрощенном виде: например, через периодическое получение статусов и отображение их в приложении. Более глубокая интеграция с логами, трассировками и действиями по регламенту может быть добавлена позже.

## 10. Техническая архитектура

На текущем этапе в качестве базовой архитектуры проекта рассматривается серверное веб-приложение на платформе .NET.

Предлагаемая технологическая основа:
- **Blazor Server** — для пользовательского интерфейса;
- **ASP.NET Core** — для серверной логики приложения;
- **PostgreSQL** — для хранения предметных данных системы;
- интеграции с наблюдательным контуром — для получения статусов, метрик и служебной информации.

Проект может быть организован по слоям:
- **Domain** — основные сущности и бизнес-правила;
- **Application** — сценарии использования и прикладная логика;
- **Infrastructure** — работа с базой данных и внешними интеграциями;
- **Web** — пользовательский интерфейс и точка входа в приложение.

Такой подход позволяет отделить бизнес-смысл системы от технической реализации и упростить дальнейшее развитие проекта.

На первом этапе допустима умеренно упрощенная архитектура, однако уже сейчас важно заложить понятное разделение ответственности между слоями приложения.

## 11. Границы версии V1

Первая версия проекта должна решить ключевую задачу: дать пользователю рабочий инструмент для просмотра сервисов, понимания их текущего состояния и выполнения базового первичного анализа.

В версию V1 рекомендуется включить:
- список сервисов;
- категории сервисов;
- базовые статусы сервисов;
- карточку сервиса;
- поиск и фильтрацию;
- пользователей и роли в упрощенном виде;
- историю изменения статусов;
- служебные комментарии;
- базовую интеграцию с источником состояния сервисов.

В версию V1 не рекомендуется включать:
- сложное автоматическое восстановление сервисов;
- широкий набор эксплуатационных действий;
- глубокую оркестрацию инцидентного процесса;
- сложные аналитические отчеты;
- большое количество внешних интеграций на старте.

Таким образом, V1 должна быть полезной, реалистичной и технически выполнимой, но не перегруженной функциями, которые можно перенести в следующие этапы.

## 12. Структура репозитория и документации

Для проекта рекомендуется разделить описание и реализацию, но не смешивать все подряд в одной общей папке. Базовая структура репозитория может быть следующей:

```text
first_app_for_bank/
├─ README.md
├─ docs/
│  ├─ 01_project_foundation.md
│  ├─ 02_data_model.md
│  ├─ 03_pages_and_navigation.md
│  ├─ 04_non_functional_requirements.md
│  ├─ 05_observability_and_incidents.md
│  └─ 06_roadmap.md
├─ src/
│  ├─ FirstAppForBank.sln
│  ├─ FirstAppForBank.Web/
│  ├─ FirstAppForBank.Application/
│  ├─ FirstAppForBank.Domain/
│  └─ FirstAppForBank.Infrastructure/
└─ tests/
   ├─ FirstAppForBank.UnitTests/
   └─ FirstAppForBank.IntegrationTests/
```

Назначение основных разделов:
- `README.md` — краткое описание проекта и точка входа;
- `docs/` — документация и проектные блоки;
- `src/` — исходный код приложения;
- `tests/` — тесты.

Такое разбиение помогает сохранить порядок в проекте: документация развивается отдельно, код — отдельно, а структура репозитория с самого начала выглядит как у реального рабочего продукта.
